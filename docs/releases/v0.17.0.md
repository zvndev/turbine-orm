# turbine-orm v0.17.0

**Release-readiness pass.** Two correctness fixes, the PII-safe error guarantee made real, honest footprint numbers, and a size gate that measures something real.

This release is **backward-compatible**. The only behavior change is that constraint-error *messages* no longer include row values by default — see [Upgrade notes](#upgrade-notes).

```bash
npm install turbine-orm@0.17.0
```

## Correctness fixes

### Studio now works on plain PostgreSQL (critical)

Every Studio data and query request issued `SET LOCAL statement_timeout = $1`. Postgres rejects bind parameters on `SET`, so each query returned a 500 with `syntax error at or near "$1"`. The CockroachDB and YugabyteDB adapters had the same flaw.

All three now use `SELECT set_config('statement_timeout', $1, true)` — the parameterizable, transaction-local form. A new integration test (`studio-timeout-integration`) runs the real SQL against a live connection, so this can't ship green again. (The original unit test mocked the pool and never sent SQL to a server, which is exactly how the bug slipped through.)

### Constraint errors no longer leak row values (critical)

Turbine's documented guarantee is that error messages are PII-safe — column and constraint *names*, never the actual user data. But `UniqueConstraintError` and friends unconditionally appended Postgres's raw `detail` string to `.message`, e.g.:

```
Key (email)=(alice@example.com) already exists.
```

That value would land in Sentry, Datadog, and your logs. Now the raw `detail` is appended only in `verbose` mode. In the default `safe` mode, the message carries column / constraint / key *names* only.

Structured fields (`.columns`, `.column`, `.constraint`) and `.cause` still expose full detail for programmatic use. This applies to E008 (`UniqueConstraintError`), E009 (`ForeignKeyError`), E010 (`NotNullViolationError`), E011 (`CheckConstraintError`), and E016 (`ExclusionConstraintError`).

### belongsTo nested writes with a NOT NULL foreign key (high)

This now works:

```ts
db.posts.create({
  data: {
    title: 'Hello',
    user: { connect: { id: 1 } }, // or create / connectOrCreate
  },
});
```

Previously, Turbine inserted the `posts` row first and set the FK with a follow-up `UPDATE` — which failed the `NOT NULL` constraint on `user_id` at the initial `INSERT`. belongsTo relations are now resolved **before** the parent INSERT and their foreign key folded directly into it. The hasMany direction is unchanged. New integration coverage exercises `connect`, `create`, and `connectOrCreate`.

## Honest footprint claims

The README and package description previously headlined "~110 KB" against "Prisma's 1.6 MB WASM." That compared a *minified bundle* against an *unpacked install* — not a fair comparison (Turbine's own unpacked install is ~1.7 MB).

Claims now lead with the true, durable differentiator — **one dependency, no WASM engine** — with correctly-labeled bundle figures: ~27 KB brotli for the main entry, ~19 KB for the edge entry.

## Smaller changes

- **`size-limit` now measures the real bundled import graph** (`@size-limit/esbuild`) instead of the 2.6 KB barrel file, so the gate guards an actual number. Limits: 35 kB main, 25 kB edge.
- **`exports` map lists `types` first** in each conditional export, so TypeScript resolves declarations correctly under every module-resolution mode.
- **`@types/pg` is now a runtime dependency.** Turbine's public `.d.ts` re-exports `pg` types, so strict consumers (`skipLibCheck: false`) no longer hit `TS7016`.
- **Docs corrected:** `SECURITY.md` supported-versions table, README error-code list (now through E016), the full-text-search note (the `search` filter shipped in 0.15), the `CONTRIBUTING.md` architecture tree, and the benchmark results (dated and version-caveated — measured on 0.7.1; the core read path is unchanged).

## Tests

- Full suite: **1127 passing** against a live database, 0 failures.
- New regression tests: belongsTo nested-write integration (3), Studio statement-timeout integration (3), PII-safe constraint-error messages (4).
- `test:coverage` now runs the full unit set (it had drifted to a stale ~20-file subset). Coverage passes: lines 78%, functions 83%, branches 86%.

## Upgrade notes

**This release is backward-compatible. No code changes are required.**

The one behavior change: **constraint-error messages no longer include row values by default.** If you relied on the raw Postgres `detail` (with values) appearing in `.message`, opt back in when constructing the client:

```ts
const db = new TurbineClient({ pool, errorMessages: 'verbose' }, schema);
```

In the default `safe` mode, the full detail is still available programmatically — read `.columns`, `.constraint`, `.where`, or `.cause` instead of parsing `.message`. See [STABILITY.md](../../STABILITY.md#error-message-text-is-not-part-of-the-contract) for why message *text* is never part of the API contract.
